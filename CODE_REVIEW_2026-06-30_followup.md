## ご指定の確認ポイント

### 1. `_hasValidAnswer()` の窓内化が LV.1〜Max に副作用を起こしていないか → 問題なし

ブルートフォースで全レベル × 全ルート位置 × 全インターバルを総当たりした結果、**全レベルで「1試行成功率 p=1.0000」**（開放弦除外 ON/OFF 両方）。

- 出題され得る全組合せが表示窓内で解ける → 副作用なし。
- 現行のレベル定義では **リトライ（最大30回）は一度も発火しない**（毎回1発で有効問題が引ける）。
- `_pickRootPosition` から冗長条件（`calcDisplayRange(fret).end <= MAX_FRET`）を削除した点も `fret <= MAX_FRET` と等価で挙動不変。

### 2. 練習モード `[0,1,2]` 拡張で R・3rd・5th すべて解決可能か → 解決を確認

再ブルートフォースで `practice` も **p=1.0000・解けない組合せ 0件**。

```
[practice] OK(全組合せ解ける) | 1試行成功率p=1.0000 | 30回全滅確率=0.00e+0
```

前回詰んでいた「C / 5th」は、5弦(A)10F の G が表示窓 `7〜12F` に入るため解決。R・3rd・5th すべて窓内に正解あり。

### 3. `suppressNextClick` 方式が新たな不具合を生んでいないか → 実用上問題なし（極めて軽微な残課題のみ）

- touchend で `preventDefault()` + フラグ立て + 処理、click 側はフラグが立っていればスキップ → **タップは必ず1回だけ処理**。二重発火は解消。
- 理論上のエッジ: iOS では `preventDefault()` により合成 click 自体が来ないため `suppressNextClick=true` が消費されず残留する。ただし次のタップも touchend を必ず経由してフラグを立て直すため実害なし。残留フラグが問題になるのは「touchend を伴わない純マウス click がハイブリッド機で割り込む」場合のみで、iPad 運用では発生しない。完全を期すなら click 側でも処理後に必ず `false` に戻す（任意）。

## 今回新たに気づいた軽微な点（前回未指摘箇所）

### 【軽微A】`_playQuestion` の 700ms タイマーが `stop()` で止まらない

- `game.js:172` の `setTimeout(…, 700)` は追跡されていない。
- 問題提示から 700ms 以内に「ホーム」「リトライ」を押すと、このタイマーが後から発火して**ターゲット音が画面遷移後に鳴り**、非練習では `_startTimer()` まで走る（中断ゲームの `_qStartTime` を再セット。結果には影響しないが鳴音が不自然）。
- **修正方針**: `_nextQuestionTimer` と同様に handle を保持し `stop()` で `clearTimeout`。発生頻度は低め。

### 【軽微B】`this._startTime` が未使用のデッドフィールド

- `game.js:30` で `null` 初期化されたまま、どこからも代入・参照なし（実計測は `_qStartTime` / `_totalTime`）。削除可能。

### 【軽微C】コメントの実態ズレ（ドキュメント nit）

- `game.js:134` の「18F以上にしか正解がない問題を避けるため最大30回リトライ」は、修正後は「表示窓内に正解がない問題を避けるため」が正確。
- かつ現行レベルでは発火しない安全網になったので、その旨を補足するとより正確。
